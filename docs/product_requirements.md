# 제품 요구 사양서 (Product Requirements Document)

본 문서는 30인 미만 소규모 사업장을 위한 설비관리시스템(CMMS)의 핵심 제품 요구사항을 정의합니다.

---

## 1. 개요
*   **시스템 명칭**: 소규모 사업장용 설비관리시스템 (CMMS - Computerized Maintenance Management System)
*   **대상 고객**: 30인 미만의 임직원을 보유한 중소 제조/기반 설비 사업장
*   **아키텍처 모델**: **Multi-tenant SaaS (Shared Database, Shared Schema)**
    *   **테넌트 격리(Tenant Isolation)**: 모든 데이터 쿼리 시 로그인한 사용자의 `company_id` 조건이 백엔드(JPA/Hibernate)에서 강제 바인딩되어 물리적으로 다른 회사의 데이터에 절대 접근할 수 없도록 강제 차단.
    *   **제어 평면(Control Plane)**: `company_id = 'SYSTEM'` 테넌트를 예약 테넌트로 지정하여 시스템 관리자 계정이 전체 회사를 모니터링하고 가입 회사를 관리하는 독립 제어 레이어로 동작.
*   **주요 목표**: 
    1. 설비 마스터 정보 및 부품 재고의 통합 관리
    2. 수작업 중심의 예방점검 및 작업지시를 디지털화하여 이력 추적성 확보
    3. 안전한 작업 수행을 위한 작업허가서 발행 및 결재 연동
    4. 편리한 UI를 통한 실시간 모니터링 및 결재함 관리

---

## 2. 모듈별 상세 요구사항

### 2.0. 공통 정책
*   **데이터 삭제**: 
    *   **헤더 테이블**: 데이터 삭제 시 물리적으로 삭제하지 않고 `삭제마크(delete_yn = 'Y')` 처리 (Soft Delete).
    *   **상세/아이템 테이블**: 상위 헤더가 삭제되거나 상세 항목이 변경될 때 물리적 삭제 처리 (Hard Delete).
*   **인쇄/출력**: 모든 상세 인쇄 레이아웃 상단 헤더에 `[회사명]`, `[출력자 이름]`, `[출력 일시 (YYYYMMDD:HHmm)]`를 공통으로 표시.
*   **자동 채번**: 예방점검번호, 지시번호, 허가번호, 결재번호, 구매요청번호(PUR), 재고전표번호(STK) 등은 `sequence` 테이블을 활용하여 자동 채번. **채번 키는 `회사-모듈-부서-년월`**(이 4개 조합마다 순번을 분리 관리), **출력 문자열은 `{모듈}-{부서}-{년월}-{순번}`** 형식 — 부서를 문자열에 포함해 **같은 회사·플랜트의 다른 부서가 같은 달 동일 모듈을 생성할 때 PK 충돌을 차단**한다. 회사는 PK·UI 컨텍스트에 이미 있어 문자열에서 제외. 처리자 부서(WO/WP/PM=담당, APR=기안자, PUR=요청자, STK=조작자)를 사용하며, 부서가 없는 사용자는 fallback `DEPT_ROOT`. 모듈 코드는 `AppModule`(MDM/EQP/INV/STK/PM/WO/WP/APR/BRD/PUR) ≤3자 통일 — 채번 prefix와 권한 키가 동일.
*   **문서 상태 코드 (공통 `DocStatus` enum)**: `T`(임시저장) / `P`(결재상신·진행) / `C`(결재완료 확정) / `S`(직접확정, 결재 우회) / `R`(반려) / `X`(취소). **구매 절차 상태(`proc_status`, 동일 enum에 확장)**: `O`(발주) / `D`(배송) / `I`(입고) / `E`(종료) — 기존 코드와 비충돌. 구매요청 문서상태는 결재 비연계라 `T`/`S`만 사용(`X` 미사용 — `T` 폐기는 소프트 삭제, `S` 이후 폐기는 절차 종료 `E`).

### 2.1. 로그인 및 인증 (Auth)
*   **인증 방식**: ID/PW 기반 로그인 후 **JWT Access Token** 발행.
*   **세션 연장**: 화면 내에서 토큰 만료 전 만료 기간 연장(토큰 갱신) 기능 제공.
*   **글로벌 상태 관리 (FE)**: 로그인 완료 시 React 전역 상태(Zustand)에 `[회사, 플랜트, 사용자ID, 사용자명, 권한(Role), 직책(Position)]`을 보관하여 화면 제어에 활용.

### 2.2. 조직 정보 관리 (MDM - Base)
*   **회사/플랜트**: 1개 회사는 복수의 플랜트(공장)를 가질 수 있음.
*   **부서**: 상하 계층형 구조(Tree Structure)를 지원하여 부모 부서를 가질 수 있음.
*   **사용자**: 특정 부서에 소속되며 직급(position)과 직책(title) 정보를 가짐.
*   **권한 및 상세**: 
    *   `multi_plant(Y/N)` 속성으로 다중 플랜트 조회 권한 부여.
    *   각 모듈 상세별 C/R/U/D/A (Y/N) 권한 매트릭스를 사용자 권한(Role)에 매핑. (A: 결재선 지정 없이 직접 확정(S) 상태로 등록하고 다음 단계로 갈 수 있는 전결/승인 권한)
    *   **사용자 권한 등급 (Roles)**: `system`, `admin`, `manager`, `user`로 분류하며 다음 정책을 적용.
        *   `system`: 시스템 총괄 관리자. **회사 코드(company) 등록 및 관리는 오직 `system` 고유 권한**임.
        *   `admin`: 회사 관리자. 소속 회사 내의 모든 조직 정보 등록/수정/조회 권한 및 전 모듈 전체 권한을 가짐.
        *   `manager`: 부서 관리자. 조직 정보(회사, 플랜트, 부서, 사용자 등)는 **조회만 가능(Read-Only)**하며, 업무 트랜잭션(예방점검, 작업지시 등)에서 **전결/확정(A) 권한**을 가짐.
        *   `user`: 일반 사용자. 업무 트랜잭션의 등록/수정/조회(C/R/U) 권한은 가지나 직접 확정(A) 권한은 없어 결재 상신을 거쳐야 함.
*   **저장소**: 부품 재고를 보관할 창고/저장소 등록 및 수정.
*   **공통코드**: `코드그룹 - 코드아이템` 2단계 구조. 코드아이템에 `법정검사여부(Y/N)` 속성 포함.

### 2.3. 마스터 정보 관리 (Master Info)
*   **설비 마스터**:
    *   설비기본정보(설비코드, 설비명, 설치위치, 설비타입, 설치일자, 작업허가대상 여부 등), 제조사정보(제조사, 스펙, 모델, S/N), 운영정보(비고, 파일 첨부) 관리.
    *   설비별 **점검 항목** 등록 (점검이름, 방법, 최소값, 최대값, 기준값, 단위).
    *   설비별 **점검 주기** 등록 (점검타입, 주기, 주기단위[일/주/월/년], 마지막점검일, 다음점검일).
    *   가로 리스트 출력 및 엑셀(CSV) 다운로드 기능 제공.
*   **재고 마스터**:
    *   재고기본정보(재고코드, 재고명, 타입, 부서, 단위), 제조사정보(제조사, 스펙, 모델, S/N), 주문관리(안전재고, 재주문재고, 리드타임), 운영정보(비고, 파일 첨부) 관리.
    *   가로 리스트 출력 및 엑셀(CSV) 다운로드 기능 제공.

### 2.4. 트랜잭션 정보 관리 (Transaction)
*   **예방점검**:
    *   '설비 마스터 점검 주기'의 `다음점검일`을 기준으로 점검 대상 설비 자동 추출.
    *   사용자가 점검 수행일자(기본 오늘, 변경 가능)와 점검자 및 항목별 점검값 입력.
    *   **결재 연계**: 임시저장(T), 결재상신(P -> 결재완료 시 C), 권한자는 직접 확정(S) 가능.
    *   상세 보고서 출력(인쇄) 및 가로 리스트 출력.
*   **작업지시**:
    *   계획 단계(Plan)와 실적 단계(Actual)로 이원화하여 관리.
    *   작업유형(고장정비, 예방보전 등), 작업자, 비용, 공수(시간/단위), 비고, 파일 첨부.
    *   예방점검과 동일한 결재 프로세스(임시저장, 결재상신-완결시 C, 직접 확정-S) 적용.
    *   상세 보고서 출력 및 가로 리스트 출력.
*   **작업허가**:
    *   작업지시와 연계되어 계획 수립. 복수 유형(화기, 밀폐, 전기, 고소, 굴착, 중량물 등) 선택 가능 (일반 작업은 필수 선택).
    *   유형별로 서로 다른 안전 체크시트 항목 제공 (JSON 포맷 관리).
    *   결재 연계 프로세스 적용 (임시저장, 결재상신-완결시 C, 직접 확정-S).
    *   출력 시 일반 표지 및 각 항목별 체크시트를 페이지 분리하여 인쇄.
*   **재고처리 (평균법 계산 및 이력 관리)**:
    *   **재고현황**: 저장소별 부품의 현재고량 및 금액 관리.
    *   **입출고**: 입고 시 **이동평균법(Moving Average)**으로 평균단가 및 금액 업데이트.
    *   **이동**: 저장소 A에서 저장소 B로 이동 시, A출고 및 B입고 이력을 각각 개별 행으로 기록하되 B입고 행에 A출고번호를 참조번호로 기록하여 묶음.
    *   **입출고장**: 건별/목록 입출고 입력 지원 및 건별 재고 입출고장 출력, 가로 현황판 제공. 결재 연계 없음.
    *   **재고마감**: 년월 단위로 각 창고/재고별 이력(입고, 출고, 이동, 조정 수량 및 금액) 마감 처리.
*   **구매요청/입고 (Procurement)**: 현장의 구매요청 → (오프라인 견적·선정·발주) → 발주/배송 절차 기록 → 입고(재고 반영) → 플랜트별 재고 가시화. **풀 구매 워크플로우(승인·견적·입찰·선정·발주서·회계)는 전산화하지 않음**(전산은 요청 기록·출력, 발주/배송 상태, 입고 재고 반영, 문서번호 기반 이력만 책임).
    *   **구매요청(현장)**: 자재·수량·입고 저장소·요청유형(공통코드 `PR_TYPE`) 지정 후 저장(`T`) → 확정(`S`, 결재 우회 직접확정; 구매요청서 출력·수기결재). 확정 후 요청 내용(자재/수량/저장소) 불변. **취소 없음** — `T` 폐기는 소프트 삭제, `S` 이후 폐기는 절차 종료(`E`)로 대체.
    *   **전표 취소(역분개) — IN/OUT 둘 다 지원**: 전표(docNo) 단위로 역분개(IN↔OUT, 같은 docNo로 묶음). IN 취소 시 PR `received_qty` 차감, OUT 취소 시 원본 단가로 정확 복원. **공통 조건: 그 거래 이후 동일 (창고, 품목)에 어떤 transaction(IN/OUT/MOVE/ADJ)도 없을 때만 가능**(이동평균 손상 방지). MOVE/ADJ는 페어링·정합성 이슈로 미지원.
    *   **발주/배송(구매자)**: 벤더·발주일·예정도착일 기록(발주 `O`) → 배송시작(`D`). 발주서는 오프라인, 시스템은 절차 상태만 기록(구매오더 문서 없음).
    *   **입고(현장)**: 기존 재고 이력(`InventoryHistory`)에 IN 전표로 반영(`I`). 부분/과소/초과·분할 입고 허용, **수량으로 완료 판정하지 않음**. 미입고분이 남아도 종료(`E`) 가능(입고 모달 '종료' 체크 또는 목록 독립 '종료' 액션).
    *   **벤더 마스터**: 거래처(상호·사업자번호·연락처·담당자·비고) 관리. (기준정보 아닌 구매 업무 도메인)
    *   **전표번호(docNo)**: 모든 입고/출고/이동/조정 이력에 단일 체계(`STK`) 전표번호 부여(방향은 `tx_type_code`로 구분), 문서번호 기반 입고증/출고증 출력. 구매 입고는 출처를 `ref_module='PUR'`/`ref_no=PR번호`로 PR에 연결.
    *   **권한**: 입고/취소는 재고 반영이므로 `STK` C(수동 입출고와 동일 통제), 구매요청 작성·확정·발주·배송·종료·벤더 마스터는 `PUR`. 권한 모듈 일람은 `AppModule` 단일 소스(`MDM/EQP/INV/STK/PM/WO/WP/APR/BRD/PUR`, ≤3자 통일).
*   **플랜트 스코핑**: 창고에 플랜트 속성 부여(공통부문=플랜트 미지정 허용). 일반 사용자는 지정 플랜트(`last_login_plant_id`)로 조회 스코프 고정, **멀티 플랜트 권한(`multi_plant=Y`) 사용자만** 플랜트 전환/전체 총괄. 현황은 플랜트→저장소로 브레이크다운. (사업장 = 플랜트, 신규 조직 엔티티 없음)

### 2.5. 결재 (Approval)
*   **결재 라인**: 기안 작성 후 결재자 및 유형(결재, 합의, 참조) 지정하여 상신.
*   **순번 관리**: 상신자(순번 0)부터 시작하여 순차적으로 결재 진행.
*   **결재함**: 상신함, 결재함(합의 포함), 참조함으로 구분 제공.
*   **결재 인쇄 양식**: 우측 상단에 4칸씩 2열 구조의 결재박스 렌더링.
    *   1열: 결재자 정보 (직책, 이름, 결재일시 `YYYYMMDDmm` 또는 서명/직인 공간)
    *   2열: 합의자 정보
    *   2열 하단: 참조자 명단 나열.

### 2.6. 게시판
*   **기능**: 공지사항 및 자유 게시글 작성. 
*   **공지 설정**: 공지(Y) 설정 시 목록 상단에 항상 고정 노출.
*   **댓글**: 게시글 하단에 단일 레벨 댓글 작성 및 삭제 기능 제공.

### 2.7. 마이페이지 (My Page)
*   **인증된 본인 정보 조회 및 일부 항목 수정 기능 제공**.
*   **수정 가능 필드**: 이메일 주소, 전화번호, 비밀번호.
    *   비밀번호 변경 시 보안을 위해 반드시 `현재 비밀번호` 검증 절차 수행.
*   **조회 전용 필드 (수정 불가)**: 회사코드, 플랜트ID, 부서ID, 로그인ID, 사용자명, 직급, 직책, 권한(Role).

## 3. 비기능적 요구사항
*   **동시성 제어**: 
    *   재고 처리 및 일련번호 채번 시 **비관적 락(Pessimistic Write Lock)**을 적용하여 다중 접근 시 데이터 꼬임 방지.
    *   데드락 예방을 위해 ID 기준 정렬 잠금 규칙 적용.
    *   커넥션 풀 방지를 위해 최대 락 대기 시간을 **3초**로 제한.
*   **보안**: 패스워드는 일방향 해시 알고리즘(BCrypt 등)으로 암호화하여 저장. API 통신은 JWT를 이용해 권한 제어.
*   **UI/UX**: 30인 미만 소규모 사업장의 작업자가 쉽게 사용할 수 있도록 직관적인 한글 대시보드와 가로형 목록 그리드, 인쇄 친화적 레이아웃 지원. Tailwind CSS v4를 이용한 고품질 다크/라이트 모드 테마 구성.

## 4. 글로벌 상태 및 인증 컨텍스트 설계 (useStore & Principal)

프론트엔드와 백엔드의 세션 동기화 및 다중 테넌트 지원을 위해 세션 컨텍스트 데이터를 다음과 같이 격리하여 보관 및 연동합니다.

### 4.1. BE Security Principal (서버 측 세션 컨텍스트)
*   **보관 목적**: API 호출마다의 테넌트 격리 및 인가 검증
*   **수집 항목**:
    *   `companyId`: 테넌트 격리 SQL 필터링 조건 자동 바인딩용 (필수)
    *   `userId`: `created_by`, `updated_by` 감사 로그 작성자 식별
    *   `userName`: 출력자 이름 등 기입용
    *   `roleId`: 백엔드 `@PreAuthorize` API 인가용 (`system`, `admin`, `manager`, `user`)
    *   `departmentId`: 사용자의 기본 소속 부서 식별
    *   `activePlantId`: 현재 요청에 유효한 선택 공장 식별 (다중 플랜트 스위칭 대응)

### 4.2. FE useStore (Zustand 전역 상태)
*   **보관 목적**: UI 렌더링 분기, 입력 폼 기본값 바인딩, 세션 만료 제어
*   **수집 항목**:
    *   `companyId` / `companyName`: 메인 로고 및 인쇄 회사명 표시
    *   `userId` / `userName`: 환영 문구 및 로그 표시
    *   `roleId`: 대분류 네비게이션 노출 여부
    *   `position` / `title`: 결재박스 내 직위/직책 표시
    *   `departmentId` / `deptName`: 신규 트랜잭션 기안 시 기본값 설정
    *   `activePlantId` / `plantName`: 헤더의 공장 드롭다운 선택값. 변경 시 API 헤더(`X-Active-Plant-Id`)에 자동 주입되어 BE로 전송.
    *   `permissions`: 모듈별 세부 C/R/U/D/A 권한 맵 (버전 노출 제어용)
    *   `accessToken` / `tokenExpiresAt`: 세션 타임아웃 남은 시간 표시 및 연장 유도
